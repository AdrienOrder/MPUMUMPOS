### 2.1.0. Анализ существующего процесса (AS-IS)

До разработки специализированного мобильного приложения управление устройством мониторинга осуществлялось следующим образом:

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     ПРОЦЕСС УПРАВЛЕНИЯ (AS-IS)                         │
│                 Управление устройством без приложения                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐                 │
│  │  Пользова-  │───>│  Bluetooth  │───>│  Устройство │                 │
│  │   тель      │    │  Terminal   │    │  HC-05/MCU  │                 │
│  └─────────────┘    └─────────────┘    └─────────────┘                 │
│       │                   │                   │                         │
│       │           ┌───────▼───────┐           │                         │
│       │           │  Ручной ввод │           │                         │
│       │           │   команд в   │           │                         │
│       │           │   HEX/тексте │           │                         │
│       │           └───────────────┘           │                         │
│       │                                       │                         │
│       │   ┌──────────────────────────────┐   │                         │
│       │   │  Список команд (из докумен-   │   │                         │
│       │   │  тации, требуется сверка):    │   │                         │
│       │   │  • SET interval 14400         │   │                         │
│       │   │  • SET start 01:00:00         │   │                         │
│       │   │  • GET data                   │   │                         │
│       │   │  • ADD sensor 1               │   │                         │
│       │   └──────────────────────────────┘   │                         │
│       │                                       │                         │
│       │   ┌──────────────────────────────┐   │                         │
│       │   │  Ответ устройства (сырые     │   │                         │
│       │   │  данные, требуется интерпре-│   │                         │

│       │   │  • INTERVAL: 14400          │   │                         │
│       │   │  • !data:2024/1/15,10:30;... │   │                         │
│       │   └──────────────────────────────┘   │                         │
│       │                                       │                         │
│       ▼                                       ▼                         │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │                     РУЧНАЯ ОБРАБОТКА                           │   │
│  │  • Копирование данных в Excel                                  │   │
│  │  • Построение графиков вручную                                 │   │
│  │  • Расшифровка кодов датчиков по документации                 │   │
│  │  • Сохранение файлов вручную                                   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│  ПРОБЛЕМЫ:                                                            │
│  ─────────                                                            │
│  ✗ Высокая вероятность ошибок при ручном вводе команд (~5%)           │
│  ✗ Необходимость сверки с документацией для каждой операции         │
│  ✗ Отсутствие автоматического сохранения данных                      │
│  ✗ Нет визуализации в реальном времени                               │
│  ✗ Ручная обработка CSV (3-5 минут на настройку устройства)         │
│  ✗ Нет мониторинга состояния датчиков                               │
└─────────────────────────────────────────────────────────────────────────┘

Рисунок 2.0 – Диаграмма AS-IS: управление устройством без специализированного приложения
```

### 2.1. Архитектура приложения

Мобильное приложение MPUMUMPOS разработано для платформы Android с использованием языка Kotlin. Приложение построено на принципах компонентной архитектуры с разделением логики представления, бизнес-логики и работы с данными.

### 2.7. Реестр команд и исходных данных

Для корректной работы приложения необходимо формальное описание всех команд управления и измеряемых параметров.

#### 2.7.1. Команды управления устройством

| № | Команда | Код (текстовый формат) | Параметры | Ожидаемый ответ | Описание |
|---|---------|------------------------|-----------|-----------------|----------|
| 1 | Установить интервал | `SET interval {minutes}` | Целое число (1–525600 минут = 365 дней) | `INTERVAL set` или `INTERVAL: {value}` | Задать периодичность измерений |
| 2 | Установить время старта | `SET start {dd:HH:MM}` | Строка формата ДД:ЧЧ:ММ | `START set` или `S: {time}` | Задать время начала сбора данных |
| 3 | Синхронизация времени | `SET time {HH:mm dd.MM.yyyy}` | Текущее время | `TIME set` или подтверждение | Синхронизация RTC устройства |
| 4 | Запрос интервала | `GET interval` | – | `INTERVAL: {minutes}` | Получить текущий интервал |
| 5 | Запрос времени старта | `GET start` | – | `START: {time}` или `S: {time}` | Получить время старта |
| 6 | Запрос текущего времени | `GET time` | – | `T: {time}` или `CURRENT: {time}` | Получить текущее время устройства |
| 7 | Скачать данные | `GET data` | – | `!info:...` затем `!data:...` затем `OK` | Передача CSV-файла с данными |
| 8 | Добавить датчик | `ADD sensor {id}` | ID от 1 до 999 | `ADD sensor {id} OK` или список датчиков | Регистрация датчика в системе |
| 9 | Удалить датчик | `DEL sensor {id}` | ID от 1 до 999 | `DEL sensor {id} OK` | Удаление датчика из системы |
| 10 | Запрос списка датчиков | `GET sensors` | – | Список `id:status` через запятую | Получить состояние всех датчиков |

#### 2.7.2. Формат данных устройства

Устройство передаёт данные в двух форматах:

**Информационная строка (!info:)**
```
!info:device_data_example=14000001TempAir:25.5,14000002Humidity:60.0;
```
Содержит перечень всех доступных параметров с ID датчиков.

**Строка данных (!data:)**
```
!data:2024/01/15,10:30;14000001TempAir:26.0,14000002Humidity:58.5;
```
Содержит timestamp и значения параметров.

#### 2.7.3. Измеряемые параметры

| Параметр | Ключ (sensorID + имя) | Единицы измерения | Типичный диапазон |
|----------|----------------------|-------------------|-------------------|
| Температура воздуха | 14000001TempAir | °C (градусы Цельсия) | -40 … +85 |
| Влажность | 14000002Humidity | % (проценты) | 0 … 100 |
| Датчик температуры 1 | {ID}TempAir | °C | -40 … +85 |
| Датчик влажности 1 | {ID}Humidity | % | 0 … 100 |

*Примечание: конкретные ID датчиков зависят от конфигурации устройства.*

### 2.9. Валидация результатов

Для подтверждения работоспособности приложения были протестированы типичные сценарии использования.

#### 2.9.1. Положительный сценарий

**Сценарий**: Полный цикл работы с устройством

1. Пользователь открывает приложение
2. Нажимает «Подключить» → выбирает устройство HC-05 из списка
3. Устанавливает интервал измерений: 10 минут (600 минут)
4. Синхронизирует время с устройством
5. Ожидает накопления данных (1 час)
6. Нажимает «Скачать данные в хранилище»
7. Открывает Storage → выбирает файл → Visualization
8. Выбирает параметры для отображения → Построить график
9. Применяет фильтр Day/Month/Year для анализа

**Результат**: Все операции выполнены успешно, график отображает корректные данные, временные метки соответствуют интервалу измерений.

#### 2.9.2. Сценарий с ошибкой ввода

**Сценарий**: Некорректный ID датчика

1. Пользователь открывает экран «Работа с датчиками устройства»
2. Нажимает «Добавить датчик»
3. Вводит ID: 9999
4. Нажимает «ПОДТВЕРДИТЬ»

**Результат**: Приложение отклоняет ввод, отображает Toast «Некорректный ID датчика». Команда не отправляется на устройство. Логирование: «Добавление датчика по ID: 9999» с последующим сообщением об ошибке.

#### 2.9.3. Сценарий с потерей соединения

**Сценарий**: Разрыв Bluetooth во время скачивания данных

1. Пользователь инициирует скачивание данных (кнопка «Скачать данные в хранилище»)
2. Устройство передаёт данные (!data:)
3. Соединение разрывается (устройство выключено)

**Результат**: Приложение фиксирует обрыв соединения:
- BluetoothManager вызывает onDisconnected callback
- LogStorageManager записывает «Устройство отключено»
- Toast отображает «Соединение с устройством потеряно»
- DataSaveHelper сохраняет уже полученные данные
- Приложение не падает, переходит в режим ожидания повторного подключения

#### 2.9.4. Выводы по валидации

Приложение устойчиво к некорректным действиям пользователя и сбоям связи:
- Валидация входных данных предотвращает отправку некорректных команд
