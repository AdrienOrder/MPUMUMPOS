# ГЛАВА 2. РАЗРАБОТКА ПРИЛОЖЕНИЯ

Данная глава посвящена описанию процесса разработки мобильного приложения для мониторинга параметров окружающей среды. Приложение разработано на языке Kotlin для платформы Android и обеспечивает полный цикл взаимодействия с устройством мониторинга: от поиска и подключения до визуализации собранных данных.

---

## 2.1. Поиск устройств и установка Bluetooth-соединения

Первым этапом работы с приложением является обнаружение и подключение к целевому устройству мониторинга. Для этого в приложении реализован механизм сканирования доступных Bluetooth устройств и установки соединения по протоколу RFCOMM.

### 2.1.1. Архитектура Bluetooth взаимодействия

Управление Bluetooth соединением в приложении осуществляется специализированным классом `BluetoothManager`, реализованным по паттерну Singleton. Данный подход обеспечивает единую точку доступа к Bluetooth функциональности из любой активности приложения и исключает возможность создания дублирующих соединений.

Класс `BluetoothManager` использует асинхронную модель взаимодействия с устройством. Все операции подключения, отправки команд и получения данных выполняются в отдельных потоках (Thread), что предотвращает блокировку пользовательского интерфейса. Для обновления UI используется Android Handler, позволяющий безопасно передавать данные из фонового потока в основной.

Ключевой особенностью реализации является механизм множественных обработчиков ответов. Статический метод `setAdditionalHandler()` позволяет любому компоненту приложения подписаться на получение всех входящих данных от устройства:

```kotlin
BluetoothManager.setAdditionalHandler { response ->
    processSensorResponse(response)
}
```

Данный механизм критически важен для параллельной работы нескольких активностей. Например, экран работы с датчиками (`SensorActivity`) должен обрабатывать ответы устройства одновременно с основной активностью (`MainActivity`), которая также ожидает данные для отображения статуса подключения.

Внутренняя реализация `BluetoothManager` включает:
- Поток чтения данных из InputStream
- Очередь команд для отправки через OutputStream
- Обработку событий подключения и отключения
- Логирование всех операций

### 2.1.2. Процесс сканирования устройств

Поиск устройств осуществляется в `DeviceListActivity` с использованием системного механизма обнаружения Bluetooth. Процесс включает следующие последовательные этапы.

На первом этапе выполняется инициализация адаптера. Приложение получает ссылку на системный BluetoothAdapter и проверяет его доступность. Если Bluetooth отключён, пользователю предлагается включить его.

На втором этапе запускается сканирование. Метод `startDiscovery()` инициирует поиск доступных устройств в радиусе действия. Система Android автоматически выполняет сканирование и генерирует события для каждого обнаруженного устройства.

На третьем этапе происходит обработка результатов. Приложение регистрирует `BroadcastReceiver` для приёма событий обнаружения новых устройств. Для каждого найденного устройства извлекаются имя и MAC-адрес. Если устройство не имеет имени, отображается строка «Неизвестное устройство».

На завершающем этапе сканирование останавливается. Это происходит либо при обнаружении целевого устройства (HC-05), либо по истечении таймаута. Остановка сканивания необходима для экономии заряда батареи устройства.

Список найденных устройств отображается с помощью `DeviceAdapter`. Адаптер формирует элементы списка, содержащие название устройства, MAC-адрес и кнопку подключения. При нажатии на кнопку начинается процесс установки соединения с выбранным устройством.

### 2.1.3. Установка соединения с устройством

Подключение к выбранному устройству выполняется по протоколу RFCOMM с использованием стандартного UUID для последовательного порта Bluetooth:

```
UUID: 00001101-0000-1000-8000-00805F9B34FB
```

Данный UUID является стандартным для устройств с последовательным портом (SPP), к которым относится модуль HC-05, используемый в устройстве мониторинга.

Процесс подключения реализован в виде асинхронной операции в отдельном потоке:

```kotlin
Thread {
    try {
        bluetoothSocket = device.createRfcommSocketToServiceRecord(HC05_UUID)
        bluetoothSocket?.connect()
        outputStream = bluetoothSocket?.outputStream
        inputStream = BufferedReader(InputStreamReader(bluetoothSocket?.inputStream))
        
        isConnected = true
        connectedDevice = device
        
        handler.post { onConnected(device) }
        startReading()
    } catch (e: IOException) {
        Log.e(TAG, "Connection error: ${e.message}")
        LogStorageManager.logMessage("Ошибка: ${e.message}")
        handler.post { onError("Ошибка подключения: ${e.message}") }
    } catch (e: SecurityException) {
        Log.e(TAG, "Security error: ${e.message}")
        LogStorageManager.logMessage("Ошибка безопасности")
        handler.post { onError("Ошибка безопасности: нет разрешений для доступа к Bluetooth") }
    }
}.start()
```

После успешного подключения выполняются следующие действия: создаются потоки ввода и вывода, устанавливается флаг соединения, вызывается callback `onConnected()` для обновления UI, запускается поток чтения данных.

Поток чтения работает в бесконечном цикле, ожидая входящие данные от устройства:

```kotlin
private fun startReading() {
    readThread = Thread {
        try {
            while (isConnected && !Thread.currentThread().isInterrupted) {
                val line = inputStream?.readLine()
                if (line != null) {
                    handler.post {
                        onMessageReceived(line)
                        additionalHandler?.invoke(line)
                    }
                }
            }
        } catch (e: IOException) {
            if (isConnected) {
                Log.d(TAG, "Соединение разорвано: ${e.message}")
                disconnect()
            }
        }
    }
    readThread?.start()
}
```

При получении данных вызываются два обработчика: основной (`onMessageReceived`) для обработки в MainActivity и дополнительный (`additionalHandler`) для параллельной обработки в других компонентах приложения.

### 2.1.4. Отправка команд на устройство

Команды отправляются через метод `sendCommand()`, который добавляет символы перевода строки к каждой команде в соответствии с протоколом устройства:

```kotlin
fun sendCommand(command: String) {
    try {
        val commandWithNewline = if (command.endsWith("\r\n") || command.endsWith("\n")) 
            command else "$command\r\n"
        Log.d(TAG, "Sent: $commandWithNewline")
        outputStream?.write(commandWithNewline.toByteArray())
        outputStream?.flush()
    } catch (e: IOException) {
        Log.e(TAG, "Send error: ${e.message}")
        LogStorageManager.logMessage("Ошибка: ${e.message}")
        handler.post { onError("Ошибка отправки: ${e.message}") }
    }
}
```

Приложение использует следующий набор команд для взаимодействия с устройством:

| Команда | Формат | Параметры | Ожидаемый ответ | Назначение |
|--------|--------|-----------|-----------------|------------|
| SET INTERVAL | `SET interval {минуты}` | Целое число (1–525600) | `INTERVAL set` | Установка интервала измерений |
| SET START | `SET start {dd:HH:MM}` | Строка формата ДД:ЧЧ:ММ | `START set` | Установка времени старта |
| SET TIME | `SET time {HH:mm dd.MM.yyyy}` | Текущее время | `TIME set` | Синхронизация времени |
| GET INTERVAL | `GET interval` | – | `INTERVAL: {value}` | Запрос текущего интервала |
| GET START | `GET start` | – | `START: {time}` | Запрос времени старта |
| GET TIME | `GET time` | – | `T: {time}` или `CURRENT: {time}` | Запрос текущего времени |
| GET DATA | `GET data` | – | `!info:` + `!data:` + `OK` | Скачивание накопленных данных |
| ADD SENSOR | `ADD sensor {id}` | ID (1–999) | Список датчиков | Добавление датчика |
| DEL SENSOR | `DEL sensor {id}` | ID (1–999) | `OK` | Удаление датчика |
| GET SENSORS | `GET sensors` | – | Список `id:status` | Запрос списка датчиков |

### 2.1.5. Обработка ответов устройства

Ответы от устройства обрабатываются в методе `processResponse()` класса `MainActivity`. Система распознаёт различные типы ответов на основе анализа текста.

При получении ответа, содержащего «START» или начинающегося с «S:», выполняется извлечение времени старта. Используется регулярное выражение для поиска датчика времени в формате «ДД:ЧЧ:ММ» или «ЧЧ:ММ:СС». Найденное время отображается в соответствующем поле интерфейса.

При получении ответа, содержащего «CURRENT», «TIME» или начинающегося с «T:», извлекается текущее время устройства и отображается в поле «Время устройства».

При получении ответа, содержащего «INTERVAL» или начинающегося с «I:», извлекается числовое значение интервала. Значение преобразуется из минут в формат ДД:ЧЧ:ММ и отображается в соответствующем поле.

Строки, начинающиеся с «!info:» или «!data:», передаются в `DataSaveHelper.processLine()` для обработки как поток данных с устройства. При получении ответа «OK» с наличием накопленных данных вызывается сохранение файлов.

Все остальные ответы, не содержащие «OK», записываются в лог для отладки и мониторинга.

### 2.1.6. Разрыв соединения и повторное подключение

При разрыве соединения (выключение устройства, выход из зоны действия, ошибка связи) выполняется автоматическая обработка:

1. Вызывается метод `disconnect()` для закрытия сокета и освобождения ресурсов
2. Устанавливается флаг `isConnected = false`
3. Вызывается callback `onDisconnected()` для обновления UI
4. Пользователю отображается Toast с сообщением «Соединение с устройством потеряно»
5. В лог записывается сообщение «Устройство отключено»

Пользователь может повторно подключиться, выбрав устройство из списка или нажав кнопку «Подключить» на главном экране, если устройство всё ещё находится в радиусе действия.

## 2.2. Построение интерфейса управления и визуализации данных

Интерфейс приложения построен на принципах Material Design с использованием стандартных компонентов Android SDK. Каждый экран реализован как отдельная активность (Activity), наследующая функциональность от AppCompatActivity для обеспечения совместимости с различными версиями Android.

### 2.2.1. Главный экран приложения (MainActivity)

Главный экран является центральным узлом приложения, обеспечивающим доступ ко всем основным функциям. Интерфейс включает три основных функциональных блока.

Блок подключения содержит кнопку переключения состояния соединения («Подключить» / «Отключить») и текстовый индикатор статуса. При нажатии кнопки приложение либо инициирует подключение к выбранному ранее устройству, либо разрывает текущее соединение.

Блок настройки параметров содержит поля отображения интервала измерений и времени старта. Оба поля доступны только для чтения в обычном режиме. При нажатии кнопки «Редактировать» поля активируются для редактирования. Валидация проверяет корректность введённых значений: интервал должен быть в диапазоне от 1 минуты до 365 дней (525600 минут), время старта должно соответствовать формату ДД:ЧЧ:ММ.

Блок операций содержит три кнопки: «Синхронизировать текущее время» для установки времени устройства равным системному времени, «Скачать данные в хранилище» для инициирования передачи накопленных данных с устройства, «Работа с датчиками устройства» для перехода к экрану управления датчиками.

Дополнительно на экране присутствуют кнопки навигации для быстрого перехода к логам операций, хранилищу данных и экрану визуализации.

### 2.2.2. Экран работы с датчиками (SensorActivity)

Данный экран предназначен для управления беспроводными датчиками, подключёнными к устройству мониторинга. Функциональность включает отображение списка датчиков, добавление новых датчиков и удаление существующих.

Отображение списка датчиков осуществляется в виде вертикального списка. Каждый элемент списка содержит три компонента: идентификатор датчика в формате «ID: {номер}», иконку статуса и кнопку удаления. Система использует три состояния статуса: подтверждён (зелёная галочка ✓), ошибка (красный крестик ✗), поиск (фиолетовый значок поиска 🔍).

Добавление датчика выполняется через диалоговое окно. При нажатии кнопки «Добавить датчик» открывается диалог с текстовым полем для ввода ID и кнопками «ПОДТВЕРДИТЬ» и «ОТМЕНА». Валидация проверяет, что ввод не пустой и ID находится в диапазоне от 1 до 999.

При подтверждении выполняется оптимистичное добавление датчика в локальный список со статусом «поиск» (🔍). Это обеспечивает мгновенную обратную связь пользователю. Параллельно отправляется команда ADD sensor на устройство.

Мониторинг статусов работает автоматически. При получении списка датчиков от устройства статусы обновляются: 2 (поиск) меняется на 1 (подтверждён) или 0 (ошибка). Если датчик находится в состоянии поиска более 15 секунд (5 попыток опроса с интервалом 3 секунды), статус автоматически меняется на ошибку. Счётчик попыток сбрасывается при получении новых данных от устройства.

Удаление датчика выполняется через диалог подтверждения. При нажатии кнопки «Удалить» отображается диалог с заголовком голубого цвета и вопросом «Удалить датчик с ID {номер}?». После подтверждения отправляется команда DEL sensor на устройство и датчик удаляется из локального списка.

Экран автоматически загружает список датчиков при открытии (задержка 300 мс для стабилизации соединения) и обновляет список после каждого действия (добавление, удаление).

### 2.2.3. Экран хранилища данных (StorageActivity)

Хранилище данных обеспечивает централизованное управление всеми файлами данных, полученными как при импорте из папки Загрузки, так и при скачивании с устройств.

Отображение организовано по принципу «устройство → файлы». Для каждого устройства отображается название, количество связанных файлов и дата последнего скачивания. Файлы, импортированные из папки Загрузки, помечаются специальным образом (device_id = 0).

В нижней части экрана отображается статистика: «Хранилище: импортировано из Загрузок: X файлов, скачано с устройств: Y файлов». Подсчёт выполняется на основе данных из базы данных SQLite.

Операции включают импорт CSV из папки Загрузки через системный файловый менеджер, удаление устройства (каскадное удаление устройства и всех связанных файлов) и удаление отдельного файла.

Все диалоги удаления используют единый стиль оформления с заголовком голубого цвета (#1976D2) для визуальной согласованности интерфейса.

### 2.2.4. Экран списка файлов (FileListActivity)

Данный экран отображает файлы конкретного устройства с возможностью их скачивания, визуализации и удаления.

Особенностью является разделение функциональности по типу файлов. Файлы с расширением .csv имеют полную функциональность: визуализация, скачивание, удаление. Файлы с расширением .txt имеют ограниченную функциональность: только скачивание и удаление, кнопка визуализации скрыта. Это обусловлено тем, что .txt файлы являются информационными и содержат метаданные устройства, а не данные измерений.

Скачивание файлов выполняется в папку `Downloads/MPUMUMPOS Downloads`. При наличии файла с таким же именем создаётся версия с индексом в скобках: «filename (1).csv».

Экран включает нижнюю панель навигации для быстрого перехода к логам операций, главному экрану и хранилищу.

### 2.2.5. Экран выбора параметров (VisualizationActivity)

Экран визуализации служит для выбора параметров, которые будут отображаться на графиках и в таблицах. При открытии выполняется загрузка CSV файла и парсинг данных.

Загрузка данных выполняется с использованием класса `CsvDataParser`. Метод `parseCsvFile()` анализирует файл и извлекает заголовки (названия параметров) и точки данных (timestamp и значения).

Выбор параметров реализован через список чекбоксов. Каждый параметр отображается с названием и может быть включён или отключён пользователем. Система `ParamBounds` сохраняет границы значений (минимальное и максимальное) для каждого параметра, что используется при фильтрации на графиках.

После выбора параметров пользователь может перейти к построению графиков (ChartActivity) или просмотру таблицы данных (TableActivity).

### 2.2.6. Экран графиков (ChartActivity)

Экран графиков обеспечивает визуализацию выбранных параметров в виде линейных графиков с использованием библиотеки MPAndroidChart.

Основные возможности включают отображение линейного графика с несколькими рядами данных (по одному на выбранный параметр), интерактивную легенду с возможностью скрытия и отображения отдельных рядов, масштабирование (pinch-to-zoom) и перемещение по графику, выбор конкретной точки для отображения точного значения.

Выбор периода реализован через диалоговое окно с тремя опциями: Day (данные за один день), Month (данные за месяц), Year (данные за год). Фильтрация выполняется по timestamp каждой точки данных.

Экспорт в PDF формирует документ с текущим графиком и сохраняет его в папку Загрузки. При построении графика в лог добавляется сообщение «График построен» с указанием количества точек и временного диапазона.

### 2.2.7. Экран таблицы данных (TableActivity)

Экран таблицы обеспечивает отображение данных в табличном виде с возможностью фильтрации по дате.

Структура таблицы включает первый столбец с датой и временем, последующие столбцы со значениями выбранных параметров. Данные организованы в виде прокручиваемой таблицы.

Фильтрация позволяет установить начальную и конечную даты периода. Данные за пределами выбранного диапазона скрываются. При изменении фильтра таблица перерисовывается с учётом нового диапазона.

При открытии экрана в лог записывается количество выбранных параметров: «Таблица: {filename}, {count} параметров».

## 2.3. Структура данных и обмен информацией с устройством

Для обеспечения надёжного хранения и обработки данных приложение использует комбинированный подход: SQLite база данных для структурированных данных и файловая система для хранения CSV файлов.

### 2.3.1. База данных SQLite

Управление базой данных осуществляется классом `StorageDatabaseManager`, наследующим функциональность от `SQLiteOpenHelper`. База данных содержит две связанные таблицы.

Таблица `devices` хранит информацию об устройствах мониторинга. Структура включает поле `id` (INTEGER PRIMARY KEY) для уникального идентификатора, поле `name` (TEXT) для названия устройства, поле `address` (TEXT) для MAC-адреса Bluetooth, поле `created_at` (INTEGER) для времени создания записи, поле `imported_at` (INTEGER) для времени импорта (значение 0 указывает на импорт из папки Загрузки).

Таблица `csv_files` хранит информацию о файлах данных. Структура включает поле `id` (INTEGER PRIMARY KEY) для уникального идентификатора, поле `device_id` (INTEGER) для ссылки на устройство (foreign key), поле `file_name` (TEXT) для имени файла, поле `file_path` (TEXT) для полного пути к файлу, поле `downloaded_at` (INTEGER) для времени скачивания, поле `line_count` (INTEGER) для количества строк в файле.

Связь между таблицами реализована как «один-ко-многим»: одно устройство может иметь множество связанных файлов. При удалении устройства автоматически удаляются все связанные файлы благодаря механизму каскадного удаления.

Основные операции включают получение списка устройств (`getDevices()`), добавление устройства (`addDevice(name, address)`), получение файлов устройства (`getCsvFilesForDevice(deviceId)`), добавление файла (`addCsvFile(...)`), удаление устройства (`deleteDevice(deviceId)`).

### 2.3.2. Парсинг CSV файлов

Класс `CsvDataParser` обеспечивает чтение и интерпретацию CSV файлов двух форматов.

Стандартный формат CSV имеет структуру с заголовком в первой строке (Date,Time,Param1,Param2,...) и данными в последующих строках. Парсинг выполняется методом `parseCsvLine()`, который корректно обрабатывает кавычки и разделители внутри полей.

Формат устройства с разделителем точка с запятой имеет иную структуру: `2024/01/15,10:00;14000001TempAir:25.5,14000002Humidity60.0;`. Дата представлена в формате `yyyy/M/d`, время в формате `H:mm` или `H:mm:ss`, данные разделены точкой с запятой, ключ параметра содержит ID датчика (числовой префикс) и имя параметра.

Для идентификации источника данных реализована функция `extractParamName()`. Например, входная строка «14000001TempAir» преобразуется в «14000001: TempAir». Логика заключается в том, что числовой префикс интерпретируется как ID датчика, а остальная часть как имя параметра.

Метод `parseDeviceDataFormat()` обрабатывает этот формат отдельно: извлекает дату и время, парсит пары «ключ:значение», преобразует значения (заменяет запятую на точку для европейского формата), формирует точки данных с timestamp.

### 2.3.3. Сохранение данных с устройства

Класс `DataSaveHelper` обеспечивает приём и сохранение данных в реальном времени при скачивании с устройства. Процесс состоит из нескольких этапов.

На первом этапе выполняется инициализация. При нажатии кнопки «Скачать данные» вызывается метод `startCollecting(deviceAddress, deviceName)`, который подготавливает внутренние структуры для накопления данных.

На втором этапе происходит обработка входящих строк. Метод `processLine(response)` анализирует каждую входящую строку: строки, начинающиеся с `!info:`, содержат информацию о параметрах, строки, начинающиеся с `!data:`, содержат данные измерений. Метод возвращает `true` если строка была обработана и `false` для остальных типов строк.

На третьем этапе обрабатывается информация о параметрах. Из строки `!info:` извлекаются ключи параметров (sensorID + ParamName) и сохраняются для использования при обработке данных. Пример: `!info:device_data_example=14000001TempAir:25.5,14000002Humidity:60.0;`

На четвёртом этапе парсятся строки данных. Аналогично формату устройства в CSV, timestamp преобразуется в миллисекунды, значения параметров извлекаются и сохраняются.

На завершающем этапе при получении ответа `OK` вызывается `saveFiles(context)` для записи данных в базу данных и файловую систему.

### 2.3.4. Модели данных

Приложение использует набор data classes Kotlin для представления данных в памяти.

Модель устройства содержит поля: id (Long), name (String), address (String), createdAt (Long), importedAt (Long). Поле importedAt со значением 0 указывает на файлы, импортированные из папки Загрузки.

Модель CSV файла содержит поля: id (Long), deviceId (Long), fileName (String), filePath (String), downloadedAt (Long), lineCount (Int).

Информация о CSV файле (результат парсинга) включает: fileName (String), headers (List<String>), dataColumns (List<String>), dataPoints (List<CsvDataPoint>), lineCount (Int).

Точка данных содержит timestamp (Long) и values (Map<String, Double>).

Границы параметра для фильтрации содержат paramName (String), minValue (Double), maxValue (Double), selectedMin (Double?), selectedMax (Double?).

### 2.3.5. Формат данных устройства

Устройство передаёт данные в текстовом формате через Bluetooth UART. Выделяют два типа передаваемых строк.

Информационная строка (префикс `!info:`) содержит перечень всех доступных параметров с примерными значениями. Используется для определения структуры данных. Пример: `!info:device_data_example=14000001TempAir:25.5,14000002Humidity:60.0;`

Строка данных (префикс `!data:`) содержит timestamp и фактические значения параметров. Пример: `!data:2024/01/15,10:30;14000001TempAir:26.0,14000002Humidity:58.5;`

Структура строки данных включает дату в формате `yyyy/M/d`, время в формате `H:mm`, разделитель `;`, параметры в формате «Ключ:Значение», разделённые запятой.

После передачи всех данных устройство отправляет ответ `OK`, сигнализирующий о завершении передачи. Приложение сохраняет полученные данные и информирует пользователя.

## 2.4. Обработка сбоев и логирование системных событий

Надёжная работа приложения обеспечивается комплексом механизмов обработки ошибок и системы логирования всех операций.

### 2.4.1. Централизованное логирование

Класс `LogStorageManager` обеспечивает единый механизм логирования всех событий в приложении. Система позволяет записывать сообщения с автоматическим добавлением timestamp, получать все логи, подписываться на обновления для实时ного отображения в UI, очищать логи.

Использование в коде: `LogStorageManager.logMessage("Подключено к устройству: ${device.name}")`. Каждая запись содержит время события и текст сообщения.

Для снижения информативности логов применены сокращения: «синхр» вместо «синхронизировано», «пар» вместо «параметров», «имп» вместо «импортировано», «устр» вместо «устройств». Это обеспечивает более компактный вывод без потери смысла.

Отображение логов выполняется в `LogDisplayActivity`. Приложение подписывается на обновления через callback и отображает новые записи в реальном времени.

### 2.4.2. Обработка ошибок Bluetooth

При работе с Bluetooth реализованы следующие механизмы обработки ошибок.

Проверка разрешений выполняется с учётом версии Android. На Android 12 и выше (SDK 31) требуются разрешения BLUETOOTH_CONNECT и BLUETOOTH_SCAN. На более ранних версиях достаточно разрешений BLUETOOTH и BLUETOOTH_ADMIN.

Обработка IOException включает логирование ошибки и информирование пользователя через callback. При разрыве соединения автоматически вызывается метод disconnect() для освобождения ресурсов.

Обработка SecurityException выполняется при отсутствии необходимых разрешений. Пользователю отображается сообщение с описанием проблемы.

При разрыве соединения система автоматически вызывает callback onDisconnected(), который обновляет UI (изменяет текст кнопки на «Подключить», сбрасывает индикатор статуса) и информирует пользователя через Toast.

### 2.4.3. Валидация входных данных

На всех этапах ввода данных выполняется проверка корректности.

Валидация интервала измерений проверяет диапазон от 1 минуты до 365 дней (525600 минут). При некорректном вводе отображается Toast «Некорректный интервал», команда не отправляется на устройство.

Валидация ID датчика проверяет диапазон от 1 до 999. При некорректном вводе отображается Toast «Некорректный ID датчика».

Валидация времени старта проверяет соответствие формату ДД:ЧЧ:ММ через регулярное выражение.

Валидация CSV файла при импорте проверяет наличие запятых (разделитель столбцов). При отсутствии отображается Toast «Это не CSV файл» и импорт отменяется.

### 2.4.4. Обработка ошибок парсинга CSV

Класс `CsvDataParser` содержит комплексную обработку ошибок. При некорректных данных строка пропускается, ошибка логируется, парсинг продолжается. Это обеспечивает устойчивость к частично повреждённым файлам.

Обработка числовых значений выполняется с учётом европейского формата. Запятая заменяется на точку перед конвертацией в число. При невозможности конвертации значение игнорируется:

```kotlin
val value = rawValue.replace(",", ".").toDoubleOrNull()
if (value != null) {
    numericValues[paramName] = value
}
```

При пустом или некорректном файле логируется сообщение «Ошибка: файл пустой» или «Ошибка: файл не найден».

### 2.4.5. Обработка ошибок базы данных

Обработка ошибок базы данных включает проверку существования файлов и записей перед выполнением операций, каскадное удаление (при удалении устройства автоматически удаляются связанные файлы), валидацию при импорте (проверка формата CSV).

При импорте файла из папки Загрузки выполняется проверка: если первая строка пустая или не содержит запятых, импорт отменяется с логированием «Ошибка импорта: выбранный файл не является CSV файлом».

### 2.4.6. Обработка состояний датчиков

Система мониторинга датчиков включает автоматическую обработку различных состояний.

Оптимистичное добавление обеспечивает мгновенную обратную связь. При добавлении датчика он сразу отображается в списке со статусом «поиск» (🔍), даже до получения подтверждения от устройства.

Таймаут поиска ограничивает время ожидания. Максимум 5 попыток опроса (около 15 секунд с интервалом 3 секунды). При превышении лимита статус меняется на ✗. Счётчик попыток сбрасывается при получении ответа от устройства.

Обработка ошибок устройства включает при ответе `ERROR` с ID — датчику ставится статус ✗, при ответе «Датчик уже добавлен» — датчик удаляется из списка (он уже существует), при ответе `OK` — логирование без изменения статуса.

### 2.4.7. Диалоговые окна подтверждения

Критические операции (удаление устройства, датчика, файла) требуют подтверждения через диалоговое окно. Все диалоги используют единый стиль с заголовком голубого цвета (#1976D2):

```kotlin
val titleView = TextView(this).apply {
    text = "Удалить датчик с ID $sensorId?"
    setTextColor(ContextCompat.getColor(this@SensorActivity, R.color.blue_500))
}

AlertDialog.Builder(this)
    .setCustomTitle(titleView)
    .setPositiveButton("ПОДТВЕРДИТЬ") { _, _ -> /* действие */ }
    .setNegativeButton("ОТМЕНА", null)
    .show()
```

При нажатии кнопки «ПОДТВЕРДИТЬ» выполняется соответствующее действие. При нажатии «ОТМЕНА» или вне диалога никакие действия не выполняются.

### 2.4.8. Устойчивость к сбоям

Приложение спроектировано с учётом устойчивости к различным сбоям.

Разрыв соединения при скачивании: частично полученные данные сохраняются, пользователь информируется о сбое. При повторном подключении можно продолжить скачивание.

Некорректные данные: повреждённые строки пропускаются, парсинг продолжается. Это обеспечивает обработку файлов с частичными повреждениями.

Недостаток памяти: при работе с большими файлами используется построчная обработка без загрузки всего файла в память.

Одновременные запросы: механизм `cancelPendingRefresh()` предотвращает наложение запросов списка датчиков. Перед отправкой нового запроса отменяется предыдущий отложенный запрос.
