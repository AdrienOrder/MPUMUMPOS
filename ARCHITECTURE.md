# Архитектура мобильного приложения MPUMUMPOS

## ГЛАВА 2. РАЗРАБОТКА ПРИЛОЖЕНИЯ

В данной главе рассматривается процесс разработки мобильного приложения для мониторинга параметров окружающей среды. Приложение предназначено для взаимодействия с устройством мониторинга через Bluetooth интерфейс, обеспечивая управление настройками устройства, сбор данных и их визуализацию в удобном для пользователя виде.

Разработка велась на языке Kotlin для платформы Android с использованием Android SDK версии 33 (Android 13). В качестве целевых устройств рассматриваются смартфоны с поддержкой Bluetooth версии 4.0 и выше.

---

### 2.1. Поиск устройств и установка Bluetooth-соединения

Первым этапом работы с приложением является обнаружение и подключение к целевому устройству мониторинга. Для этого в приложении реализован механизм сканирования доступных Bluetooth устройств и установки соединения по протоколу RFCOMM.

#### 2.1.1. Архитектура Bluetooth взаимодействия

Управление Bluetooth соединением в приложении осуществляется специализированным классом `BluetoothManager`, реализованным по паттерну Singleton. Данный подход обеспечивает единую точку доступа к Bluetooth функциональности из любой активности приложения и исключает возможность создания дублирующих соединений.

Класс `BluetoothManager` использует асинхронную модель взаимодействия с устройством. Все операции подключения, отправки команд и получения данных выполняются в отдельных потоках (Thread), что предотвращает блокировку пользовательского интерфейса. Для обновления UI используется Android Handler, позволяющий безопасно передавать данные из фонового потока в основной.

Особенностью реализации является механизм множественных обработчиков ответов (`additionalHandler`). Статический метод `setAdditionalHandler()` позволяет любому компоненту приложения подписаться на получение всех входящих данных от устройства. Это критически важно для экрана работы с датчиками (`SensorActivity`), который должен обрабатывать ответы устройства параллельно с основной активностью (`MainActivity`).

```kotlin
// Пример установки дополнительного обработчика
BluetoothManager.setAdditionalHandler { response ->
    processSensorResponse(response)
}
```

#### 2.1.2. Процесс сканирования устройств

Поиск устройств осуществляется в `DeviceListActivity` с использованием системного механизма обнаружения Bluetooth. Процесс включает следующие этапы:

1. **Инициализация адаптера**: Получение ссылки на системный BluetoothAdapter и проверка его доступности.

2. **Запуск сканирования**: Вызов метода `startDiscovery()` для поиска доступных устройств в радиусе действия.

3. **Обработка результатов**: Регистрация `BroadcastReceiver` для приёма событий обнаружения новых устройств. Каждое найденное устройство добавляется в список с отображением имени и MAC-адреса.

4. **Остановка сканирования**: После обнаружения устройства или по таймауту сканирование останавливается для экономии энергии.

Список найденных устройств отображается с помощью `DeviceAdapter`, который формирует элементы списка с названием устройства, MAC-адресом и кнопкой подключения.

#### 2.1.3. Установка соединения

Подключение к выбранному устройству выполняется по протоколу RFCOMM с использованием стандартного UUID для последовательного порта:

```
UUID: 00001101-0000-1000-8000-00805F9B34FB
```

Процесс подключения реализован следующим образом:

```kotlin
Thread {
    try {
        bluetoothSocket = device.createRfcommSocketToServiceRecord(HC05_UUID)
        bluetoothSocket?.connect()
        outputStream = bluetoothSocket?.outputStream
        inputStream = BufferedReader(InputStreamReader(bluetoothSocket?.inputStream))
        
        isConnected = true
        handler.post { onConnected(device) }
        startReading()
    } catch (e: IOException) {
        handler.post { onError("Ошибка подключения: ${e.message}") }
    }
}.start()
```

После успешного подключения запускается поток чтения данных (`startReading()`), который в бесконечном цикле ожидает входящие данные от устройства. Полученные данные передаются двум обработчикам:
- Основному (`onMessageReceived`) — для обработки в MainActivity
- Дополнительному (`additionalHandler`) — для параллельной обработки в других компонентах

#### 2.1.4. Отправка команд на устройство

Команды отправляются через метод `sendCommand()`, который добавляет символы перевода строки (`\r\n`) к каждой команде:

```kotlin
fun sendCommand(command: String) {
    val commandWithNewline = if (command.endsWith("\r\n") || command.endsWith("\n")) 
        command else "$command\r\n"
    outputStream?.write(commandWithNewline.toByteArray())
    outputStream?.flush()
}
```

Приложение использует следующий набор команд для взаимодействия с устройством:

| Команда | Формат | Назначение |
|---------|--------|------------|
| SET INTERVAL | `SET interval {минуты}` | Установка интервала измерений |
| SET START | `SET start {dd:HH:MM}` | Установка времени старта |
| SET TIME | `SET time {HH:mm dd.MM.yyyy}` | Синхронизация времени |
| GET INTERVAL | `GET interval` | Запрос текущего интервала |
| GET START | `GET start` | Запрос времени старта |
| GET TIME | `GET time` | Запрос текущего времени |
| GET DATA | `GET data` | Скачивание накопленных данных |
| ADD SENSOR | `ADD sensor {id}` | Добавление датчика |
| DEL SENSOR | `DEL sensor {id}` | Удаление датчика |
| GET SENSORS | `GET sensors` | Запрос списка датчиков |

#### 2.1.5. Обработка ответов устройства

Ответы от устройства обрабатываются в методе `processResponse()` класса `MainActivity`. Система распознаёт различные типы ответов:

1. **Время старта** (`START`, `S:`) — регулярное выражение извлекает время из ответа
2. **Текущее время** (`CURRENT`, `TIME`, `T:`) — аналогично, для отображения времени устройства
3. **Интервал** (`INTERVAL`, `I:`) — извлечение числового значения интервала
4. **Данные** (`!info:`, `!data:`) — передача в DataSaveHelper для обработки
5. **Подтверждение** (`OK`) — сохранение данных после завершения переда

Каждый тип ответа обрабатывается отдельно, результаты отображаются в соответствующих полях интерфейса или сохраняются в базу данных.

### 2.2. Построение интерфейса управления и визуализации данных

Интерфейс приложения построен на принципах Material Design с использованием стандартных компонентов Android. Каждый экран реализован как отдельная активность (Activity), наследующая функциональность от `AppCompatActivity`.

#### 2.2.1. Главный экран (MainActivity)

Главный экран является центральным узлом приложения, обеспечивающим доступ ко всем основным функциям. Интерфейс включает следующие функциональные блоки:

**Блок подключения**:
- Кнопка «Подключить/Отключить» — переключение состояния соединения
- Индикатор статуса подключения (текст и визуальное состояние кнопки)

**Блок настройки параметров**:
- Поле ввода интервала измерений (формат ДД:ЧЧ:ММ)
- Поле ввода времени старта (формат ДД:ЧЧ:ММ)
- Кнопка «Редактировать» для активации полей ввода

**Блок операций**:
- Кнопка «Синхронизировать текущее время» — установка времени устройства равным текущему системному времени
- Кнопка «Скачать данные в хранилище» — инициирование передачи накопленных данных с устройства
- Кнопка «Работа с датчиками устройства» — переход к экрану управления датчиками

**Блок навигации**:
- Кнопка перехода к логам операций
- Кнопка перехода в хранилище данных
- Кнопка перехода к визуализации

Редактирование параметров защищено механизмом валидации. При нажатии кнопки «Редактировать» поля активируются, пользователь вносит изменения и подтверждает их кнопкой «Подтвердить изменения». Приложение проверяет корректность введённых значений:
- Интервал должен быть в диапазоне от 1 минуты до 365 дней (525600 минут)
- Время старта должно соответствовать формату ДД:ЧЧ:ММ
- При некорректных данных отображается Toast с описанием ошибки

#### 2.2.2. Экран работы с датчиками (SensorActivity)

Данный экран предназначен для управления беспроводными датчиками, подключёнными к устройству мониторинга. Функциональность включает:

**Отображение списка датчиков**:
Каждый датчик отображается в виде строки с тремя элементами:
- Идентификатор датчика (ID)
- Иконка статуса:
  - ✓ (зелёная) — датчик подтверждён и активен
  - ✗ (красная) — датчик не отвечает или ошибка
  - 🔍 (фиолетовая) — поиск датчика (временный статус при добавлении)
- Кнопка «Удалить» для удаления датчика из системы

**Добавление датчика**:
При нажатии кнопки «Добавить датчик» открывается диалоговое окно с полем ввода ID. Валидация проверяет:
- Ввод должен быть непустым
- ID должен быть числом в диапазоне от 1 до 999

После подтверждения выполняется оптимистичное добавление датчика в список со статусом «поиск» (🔍), параллельно отправляется команда `ADD sensor {id}` на устройство.

**Мониторинг статусов**:
Система автоматически отслеживает состояние датчиков:
- При получении списка датчиков от устройства статусы обновляются (2→1 или 2→0)
- Если датчик находится в состоянии «поиск» более 15 секунд (5 попыток опроса), статус автоматически меняется на ✗
- Периодичность опроса — каждые 3 секунды при наличии датчиков со статусом поиска

**Удаление датчика**:
При нажатии кнопки «Удалить» отображается диалог подтверждения с голубым заголовком. После подтверждения отправляется команда `DEL sensor {id}` и датчик удаляется из локального списка.

#### 2.2.3. Экран хранилища (StorageActivity)

Хранилище данных обеспечивает централизованное управление всеми файлами данных, полученными как при импорте из папки Загрузки, так и при скачивании с устройств.

**Отображение устройств**:
Список организован по принципу «устройство → файлы». Для каждого устройства отображается:
- Название устройства
- Количество связанных файлов
- Дата последнего скачивания

Для импортированных файлов (из папки Загрузки) отображается специальная метка.

**Счётчики файлов**:
В нижней части экрана отображается статистика:
```
Хранилище: импортировано из Загрузок: X файлов, скачано с устройств: Y файлов
```

**Операции**:
- Импорт CSV — выбор файла из папки Загрузки через системный файловый менеджер
- Удаление устройства — каскадное удаление (устройство + все связанные файлы)
- Удаление отдельного файла

**Диалоги удаления**:
Все диалоги удаления используют единый стиль оформления с заголовком голубого цвета (#1976D2) для визуальной согласованности интерфейса.

#### 2.2.4. Экран списка файлов (FileListActivity)

Отображает файлы конкретного устройства с возможностью их скачивания, визуализации и удаления.

**Особенности обработки файлов**:
- Файлы `.csv` — полная функциональность (визуализация, скачивание, удаление)
- Файлы `.txt` — только скачивание и удаление, кнопка визуализации скрыта

Это разделение обусловлено тем, что `.txt` файлы являются информационными (содержат метаданные устройства) и не требуют визуализации в виде графиков.

**Скачивание файлов**:
Файлы копируются в папку `Downloads/MPUMUMPOS Downloads`. При наличии файла с таким же именем создаётся версия с индексом в скобках.

**Навигация**:
Экран включает нижнюю панель навигации для быстрого перехода к логам, главному экрану и хранилищу.

#### 2.2.5. Экран визуализации (VisualizationActivity)

Экран выбора параметров для отображения. Загружает данные из выбранного CSV файла и предоставляет пользователю возможность отметить интересующие параметры.

**Загрузка данных**:
Парсинг CSV файла выполняется классом `CsvDataParser`, который поддерживает два формата:
1. Стандартный CSV (столбцы Date, Time, Param1, Param2, ...)
2. Формат устройства с разделителем точка с запятой (Дата,Время;Параметр1:Значение1;Параметр2:Значение2;...)

При загрузке выполняется анализ заголовков и формируется список доступных параметров.

**Выбор параметров**:
Каждый параметр отображается в виде чекбокса с названием. Система `ParamBounds` сохраняет границы значений для каждого параметра.

**Переход к визуализации**:
После выбора параметров пользователь может перейти к:
- Построению графиков (`ChartActivity`)
- Просмотру таблицы данных (`TableActivity`)

#### 2.2.6. Экран графиков (ChartActivity)

Построение и отображение графиков выбранных параметров с использованием библиотеки MPAndroidChart.

**Основные возможности**:
- Линейный график с несколькими рядами данных (по одному на параметр)
- Интерактивная легенда с возможностью скрытия/отображения рядов
- Масштабирование (pinch-to-zoom) и перемещение по графику
- Выбор конкретной точки для отображения точного значения

**Выбор периода**:
Диалог выбора периода позволяет фильтровать данные по временному диапазону:
- Day — данные за один день
- Month — данные за месяц
- Year — данные за год

Фильтрация выполняется по timestamp каждой точки данных.

**Экспорт в PDF**:
Кнопка «Экспорт в PDF» формирует документ с текущим графиком и сохраняет его в папку Загрузки.

**Логирование**:
При построении графика в лог добавляется сообщение «График построен» с указанием количества точек и временного диапазона.

#### 2.2.7. Экран таблицы (TableActivity)

Табличное отображение данных с возможностью фильтрации по дате.

**Структура таблицы**:
- Первый столбец — дата и время
- Последующие столбцы — значения выбранных параметров

**Фильтрация**:
Пользователь может установить начальную и конечную даты периода. Данные за пределами выбранного диапазона скрываются.

**Логирование**:
При открытии экрана логируется количество выбранных параметров: «Таблица: {filename}, {count} параметров»

### 2.3. Структура данных и обмен информацией с устройством

Для обеспечения надёжного хранения и обработки данных приложение использует комбинированный подход: SQLite база данных для структурированных данных и файловая система для хранения CSV файлов.

#### 2.3.1. База данных SQLite

Управление базой данных осуществляется классом `StorageDatabaseManager`, наследующим функциональность от `SQLiteOpenHelper`.

**Схема базы данных**:

Таблица `devices` — хранит информацию об устройствах мониторинга:

| Поле | Тип | Описание |
|------|-----|-----------|
| id | INTEGER PRIMARY KEY | Уникальный идентификатор устройства |
| name | TEXT | Название устройства |
| address | TEXT | MAC-адрес Bluetooth |
| created_at | INTEGER | Время создания записи (timestamp) |
| imported_at | INTEGER | Время импорта (0 = импорт из папки Загрузки) |

Таблица `csv_files` — хранит информацию о файлах данных:

| Поле | Тип | Описание |
|------|-----|-----------|
| id | INTEGER PRIMARY KEY | Уникальный идентификатор файла |
| device_id | INTEGER | Ссылка на устройство (foreign key) |
| file_name | TEXT | Имя файла |
| file_path | TEXT | Полный путь к файлу |
| downloaded_at | INTEGER | Время скачивания (timestamp) |
| line_count | INTEGER | Количество строк в файле |

**Основные операции**:

```kotlin
// Получение списка устройств
fun getDevices(): List<Device> { ... }

// Добавление устройства
fun addDevice(name: String, address: String): Long { ... }

// Получение файлов устройства
fun getCsvFilesForDevice(deviceId: Long): List<CsvFile> { ... }

// Добавление файла
fun addCsvFile(deviceId: Long, fileName: String, filePath: String, 
               downloadedAt: Long, lineCount: Int): Long { ... }

// Удаление устройства (каскадное удаление файлов)
fun deleteDevice(deviceId: Long) { ... }
```

При удалении устройства автоматически удаляются все связанные файлы благодаря механизму каскадного удаления.

#### 2.3.2. Парсинг CSV файлов

Класс `CsvDataParser` обеспечивает чтение и интерпретацию CSV файлов двух форматов.

**Стандартный формат CSV**:
```
Date,Time,TempAir,Humidity
15.01.2024,10:00,25.5,60
15.01.2024,10:30,25.8,59
```

Парсинг выполняется методом `parseCsvLine()`, который корректно обрабатывает кавычки и разделители внутри полей.

**Формат устройства (с разделителем точка с запятой)**:
```
2024/01/15,10:00;14000001TempAir:25.5,14000002Humidity:60.0;
2024/01/15,10:30;14000001TempAir:25.8,14000002Humidity:59.5;
```

Особенности этого формата:
- Дата в формате `yyyy/M/d`
- Время в формате `H:mm` или `H:mm:ss`
- Данные разделены точкой с запятой
- Ключ параметра содержит ID датчика (числовой префикс) и имя параметра

**Функция извлечения имени параметра**:
Для идентификации источника данных реализована функция `extractParamName()`:
- Вход: `14000001TempAir`
- Выход: `14000001: TempAir`

Логика: числовой префикс интерпретируется как ID датчика, остаток — как имя параметра.

#### 2.3.3. Сохранение данных с устройства

Класс `DataSaveHelper` обеспечивает приём и сохранение данных в реальном времени при скачивании с устройства.

**Процесс сбора данных**:

1. **Инициализация**: При нажатии кнопки «Скачать данные» вызывается `startCollecting(deviceAddress, deviceName)`

2. **Обработка строк**: Метод `processLine(response)` анализирует каждую входящую строку:
   - `!info:device_data_example=...` — информация о параметрах
   - `!data:...` — строка данных
   - Возвращает `true` если строка обработана, `false` для остальных

3. **Информация о параметрах**: Из строки `!info:` извлекаются ключи параметров (sensorID + ParamName) и сохраняются для использования при обработке данных.

4. **Строки данных**: Парсятся аналогично формату устройства, timestamp преобразуется в миллисекунды.

5. **Завершение**: При получении ответа `OK` вызывается `saveFiles(context)` для записи данных в базу и файловую систему.

**Формат информации о параметрах**:
```
!info:device_data_example=14000001TempAir:25.5,14000002Humidity:60.0;
```

Ключ формируется как `sensorID + ParamName` (например, `14000001TempAir`), значение может содержать запятую как десятичный разделитель.

#### 2.3.5. Формат данных устройства

Устройство передаёт данные в текстовом формате через Bluetooth UART:

**Информационная строка** (`!info:`):
Содержит перечень всех доступных параметров с примерными значениями. Используется для определения структуры данных.

```
!info:device_data_example=14000001TempAir:25.5,14000002Humidity:60.0;
```

**Строка данных** (`!data:`):
Содержит timestamp и фактические значения параметров.

```
!data:2024/01/15,10:30;14000001TempAir:26.0,14000002Humidity:58.5;
```

Структура:
- Дата (`yyyy/M/d`)
- Время (`H:mm`)
- Разделитель (`;`)
- Параметры в формате `Ключ:Значение`, разделённые запятой

**Подтверждение завершения**:
После передачи всех данных устройство отправляет `OK`, что сигнализирует о завершении передачи.

### 2.4. Обработка сбоев и логирование системных событий

Надёжная работа приложения обеспечивается комплексом механизмов обработки ошибок и системы логирования всех операций.

#### 2.4.1. Централизованное логирование

Класс `LogStorageManager` обеспечивает единый механизм логирования всех событий в приложении.

**Функциональность**:
- Добавление записи с автоматическим добавлением timestamp
- Получение всех логов
- Подписка на обновления (callback для UI)
- Очистка логов

**Использование**:
```kotlin
LogStorageManager.logMessage("Подключено к устройству: ${device.name}")
```

Логи отображаются в `LogDisplayActivity` в реальном времени. Каждая запись содержит время события и текст сообщения.

**Оптимизация**:
Для сниженияverbosity логов применены сокращения:
- «синхр» вместо «синхронизировано»
- «пар» вместо «параметров»
- «имп» вместо «импортировано»
- «устр» вместо «устройств»

#### 2.4.2. Обработка ошибок Bluetooth

При работе с Bluetooth реализованы следующие механизмы:

**Проверка разрешений**:
```kotlin
val hasBluetoothPermissions = if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
    ContextCompat.checkSelfPermission(this, Manifest.permission.BLUETOOTH_CONNECT) == 
        PackageManager.PERMISSION_GRANTED
} else {
    ContextCompat.checkSelfPermission(this, Manifest.permission.BLUETOOTH) == 
        PackageManager.PERMISSION_GRANTED
}
```

**Обработка IOException**:
```kotlin
catch (e: IOException) {
    Log.e(TAG, "Connection error: ${e.message}")
    LogStorageManager.logMessage("Ошибка: ${e.message}")
    handler.post { onError("Ошибка подключения: ${e.message}") }
    disconnect()
}
```

**Обработка разрыва соединения**:
```kotlin
catch (e: IOException) {
    if (isConnected) {
        Log.d(TAG, "Соединение разорвано: ${e.message}")
        disconnect()
    }
}
```

При разрыве соединения автоматически вызывается callback `onDisconnected()`, который обновляет UI и информирует пользователя.

#### 2.4.3. Валидация входных данных

На всех этапах ввода данных выполняется проверка корректности:

**Интервал измерений**:
- Диапазон: от 1 минуты до 365 дней (525600 минут)
- Сообщение при ошибке: «Некорректный интервал»

**ID датчика**:
- Диапазон: от 1 до 999
- Сообщение при ошибке: «Некорректный ID датчика»

**Время старта**:
- Формат: ДД:ЧЧ:ММ
- Проверка через регулярное выражение

**CSV файл при импорте**:
- Проверка наличия запятых (разделитель столбцов)
- Сообщение: «Это не CSV файл»

#### 2.4.4. Обработка ошибок парсинга CSV

Класс `CsvDataParser` содержит комплексную обработку ошибок:

```kotlin
try {
    // Парсинг файла
} catch (e: Exception) {
    LogStorageManager.logMessage("Ошибка: ${e.message}")
    return null
}
```

При некорректных данных строка пропускается, ошибка логируется, парсинг продолжается. Это обеспечивает устойчивость к частично повреждённым файлам.

**Обработка числовых значений**:
```kotlin
val value = rawValue.replace(",", ".").toDoubleOrNull()
if (value != null) {
    numericValues[paramName] = value
}
```

Запятая заменяется на точку (европейский формат), при невозможности конвертации значение игнорируется.

#### 2.4.5. Обработка ошибок базы данных

**Проверка существования**:
```kotlin
if (sourceFile.exists()) {
```

**Каскадное удаление**:
При удалении устройства автоматически удаляются связанные файлы из базы данных.


#### 2.4.6. Обработка состояний датчиков

Система мониторинга датчиков включает автоматическую обработку различных состояний:

**Оптимистичное добавление**:
При добавлении датчика он сразу отображается в списке со статусом «поиск» (🔍), что обеспечивает мгновенную обратную связь пользователю.

**Таймаут поиска**:
- Максимум 5 попыток опроса (около 15 секунд)
- При превышении лимита статус меняется на ✗
- Счётчик попыток сбрасывается при получении ответа от устройства

**Обработка ошибок устройства**:
- При ответе `ERROR` с ID — датчику ставится статус ✗
- При ответе «Датчик уже добавлен» — датчик удаляется из списка
- При ответе `OK` — логирование без изменения статуса

#### 2.4.7. Диалоговые окна подтверждения

Все диалоги используют единый стиль с заголовком голубого цвета (#1976D2).

#### 2.4.8. Устойчивость к сбоям

Приложение спроектировано с учётом устойчивости к различным сбоям:

1. **Разрыв соединения при скачивании**: Частично полученные данные сохраняются, пользователь информируется о сбое

2. **Некорректные данные**: Повреждённые строки пропускаются, парсинг продолжается

3. **Недостаток памяти**: При работе с большими файлами используется построчная обработка без загрузки всего файла в память

4. **Одновременные запросы**: Механизм `cancelPendingRefresh()` предотвращает наложение запросов списка датчиков
